# RouterLang

**A domain-specific language for declarative network topology and routing policy.**

RouterLang is a compiled DSL for ISP and enterprise backbone environments. Instead of writing vendor-specific configuration files by hand, RouterLang lets network engineers describe their entire infrastructure declaratively — topology, protocols, routing policy, and traffic intent — and compile it into validated, per-device configuration files ready for deployment.

```
topology { ... }   routing { ... }   policy { ... }   intent { ... }
         │                │               │                │
         └────────────────┴───────────────┴────────────────┘
                              Compiler
                                 │
              ┌──────────────────┼──────────────────┐
         R-SPINE-1.cfg      R-LEAF-1.cfg       R-EDGE-1.cfg
```

## What Problem It Solves

Configuring a real network means writing hundreds of lines of vendor-specific CLI commands — once per device, by hand. A small topology change can require touching every router in the network, with no compile-time safety net. Manual configuration is slow, error-prone, and produces no audit trail between intent and implementation.

RouterLang solves this by separating **intent** from **implementation**:

1. Describe the network once, at a high level, in a single `.rl` source file.
2. RouterLang validates it — catching 14 categories of structural and semantic errors before anything touches a live device.
3. RouterLang generates one configuration file per device, in the vendor syntax of your choice.

**Measured impact on an 8-device fat-tree topology:**

| Metric | RouterLang | Manual |
|---|---|---|
| Source lines authored | 87 | 520 |
| Config lines generated | 520 | 520 |
| Pre-deployment errors detected | 0 | 3 |
| Time to configure (estimate) | < 5 min | ~4 hours |
| Topology change (add 2 leaf nodes) | 3 lines changed | Full reconfiguration |

## How It Works

A `.rl` source file passes through a six-stage pipeline. The first four stages always run. Stage 5 runs when you pass `--generate`. Stage 6 runs when you also pass `--deploy`.

```
Source (.rl)
    │
    ├─ Stage 1 — Lexer           tokenises into 38 token types (ANTLR4)
    ├─ Stage 2 — Parser          builds the parse tree — 41 grammar productions (ANTLR4)
    ├─ Stage 3 — Symbol Table    collects all declared roles, links, policies, intents
    ├─ Stage 4 — Semantic Check  enforces 14 validation rules
    ├─ Stage 5 — Config Gen      emits one .cfg / .conf / .json per device  [--generate]
    └─ Stage 6 — Deploy          pushes configs to devices via NAPALM        [--deploy]
```

Config generation and deployment are intentionally decoupled. You can generate and review all output files before any change reaches a live device.

## Installation

**Prerequisites:**
- Python 3.12 or later
- Java 11 or later *(only needed to regenerate the parser after grammar changes — not required for normal use)*

**Step 1 — Clone the repository**

```bash
git clone https://github.com/EftodeAndreiFAF242/RouterLang.git
cd RouterLang
```

**Step 2 — Install runtime dependencies**

```bash
pip install antlr4-python3-runtime==4.13.1
```

For deployment to real devices, also install NAPALM:

```bash
pip install napalm
```

**Step 3 — (Optional) Regenerate the parser after grammar changes**

If you modify `src/grammar/RouterLang.g4`:

```bash
java -jar antlr-4.13.1-complete.jar -Dlanguage=Python3 -visitor -o src/parser src/grammar/RouterLang.g4
```

## Usage

### Validate a source file

Runs all four validation stages and reports errors and warnings. No files are written.

```bash
python main.py my_network.rl
```

### Validate and generate device configuration files

```bash
python main.py my_network.rl --generate
```

Writes one config file per device to `./output/my_network/` by default.

### Generate for a specific vendor

The same `.rl` source file compiles to all three vendor formats with no changes to the source:

```bash
python main.py my_network.rl --generate --vendor cisco       # Cisco IOS CLI  (default, .cfg)
python main.py my_network.rl --generate --vendor junos       # JunOS hierarchical CLI  (.conf)
python main.py my_network.rl --generate --vendor openconfig  # OpenConfig JSON  (.json)
```

### Use real IP addresses from an IPAM file

Provide a CSV file mapping device names to real IP addresses. Columns must be `device`, `loopback_ip`, `mgmt_ip`:

```csv
device,loopback_ip,mgmt_ip
R-SPINE-1,10.0.1.1,192.168.1.1
R-LEAF-1,10.0.2.1,192.168.1.11
```

```bash
python main.py my_network.rl --generate --ipam ipam.csv
```

If a device is not found in the CSV, or no `--ipam` flag is provided, the generator falls back to deterministic hash-derived IPs automatically.

### Deploy generated configs to devices

```bash
# Dry run — show diffs, apply nothing
python main.py my_network.rl --generate --deploy --dry-run --ipam ipam.csv

# Live deploy via NAPALM SSH
python main.py my_network.rl --generate --deploy --ipam ipam.csv

# With a custom credentials file
python main.py my_network.rl --generate --deploy --creds credentials.ini --ipam ipam.csv
```

Deployment requires an `--ipam` file with `mgmt_ip` addresses. A deploy log is written to `./logs/` after every run. OpenConfig JSON cannot be deployed via NAPALM — use NETCONF or RESTCONF for that format.

### Run the built-in example

```bash
python main.py --example --generate
python main.py --example --generate --vendor openconfig
```

### Additional flags

| Flag | Effect |
|---|---|
| `--tokens` | Print the full token stream after lexing |
| `--symbols` | Print the symbol table after parsing |
| `--verbose` | Enable both `--tokens` and `--symbols` |
| `--generate` | Run Stage 5 — emit per-device config files |
| `--vendor <name>` | Target vendor: `cisco` (default), `junos`, `openconfig` |
| `--out-dir <path>` | Custom output directory for generated configs |
| `--ipam <path>` | CSV file mapping devices to real IP addresses |
| `--deploy` | Run Stage 6 — push configs to devices via NAPALM |
| `--dry-run` | Used with `--deploy`: show diffs only, apply nothing |
| `--creds <path>` | Path to `credentials.ini` (default: `credentials.ini`) |

## Language Reference

### Program Structure

A RouterLang program consists of up to five top-level blocks in a fixed order:

```
topology   { ... }   -- required
routing    { ... }   -- required
policy     { ... }   -- required
intent     { ... }   -- required
transition { ... }   -- optional
```

Every block is required except `transition`. The order is fixed — the compiler rejects any program that deviates from it. Blocks are delimited by `{ }`. There are no semicolons. Whitespace and indentation are not significant.

---

### Topology Block

The topology block is always first. It defines the logical structure of the network: the roles devices play, how those roles connect, and which physical devices fill each role.

```
topology {
  roles {
    <role-name>  { count: <N> }
    <role-name>  { count: <min>..<max> }
  }
  links {
    <role> -- <role>  { weight: <N> }
  }
  devices {
    <role>: [<device-name>, <device-name>, ...]
  }
}
```

#### roles

Roles are named logical categories — `spine`, `leaf`, `edge`, `core`, `border`, etc. Every other block in the program refers to roles by name, so all roles must be declared here first.

The `count` field specifies how many physical devices fill this role. It accepts an exact integer or a range:

```
spine  { count: 2 }       -- exactly 2 spine devices
leaf   { count: 1..8 }    -- between 1 and 8 leaf devices
```

Avoid role names that collide with reserved keywords: `all`, `border`, `auto`, `area`, `route`, `scope`, `any`, `permit`, `deny`, `rank`, `match`, `set`, `from`, `to`.

#### links

Links declare which roles are physically connected. Links are undirected — `spine -- leaf` and `leaf -- spine` are equivalent. Self-links (`spine -- spine`) are not permitted (E-6).

The optional `weight` attribute controls routing preference. Lower weight is preferred; the default is 1.

```
links {
  spine -- leaf  { weight: 1 }    -- preferred path
  spine -- edge  { weight: 2 }
  edge  -- leaf  { weight: 3 }    -- least preferred
}
```

Links declared here are used by the intent block to validate path expressions. If an intent declares `edge >> spine >> leaf`, the compiler checks that `edge -- spine` and `spine -- leaf` links both exist (E-11).

#### devices

Binds concrete router hostnames to their role. This section is optional — if omitted, the config generator synthesises names automatically (e.g. `R-SPINE-1`, `R-SPINE-2`).

```
devices {
  spine: [R-SPINE-1, R-SPINE-2]
  leaf:  [R-LEAF-1, R-LEAF-2, R-LEAF-3, R-LEAF-4]
  edge:  [R-EDGE-1, R-EDGE-2]
}
```

Device names must be unique across the program and may contain letters, digits, and hyphens.

**Complete example:**

```
topology {
  roles {
    spine  { count: 2 }
    leaf   { count: 4 }
    edge   { count: 2 }
  }
  links {
    spine -- leaf  { weight: 1 }
    spine -- edge  { weight: 2 }
    edge  -- leaf  { weight: 3 }
  }
  devices {
    spine: [R-SPINE-1, R-SPINE-2]
    leaf:  [R-LEAF-1, R-LEAF-2, R-LEAF-3, R-LEAF-4]
    edge:  [R-EDGE-1, R-EDGE-2]
  }
}
```

### Routing Block

The routing block configures BGP and OSPF. It must contain at least one of the two protocol sections.

```
routing {
  bgp {
    asn {
      <role>: <ASN>
    }
    neighbors: auto
    route-reflector: <role>
  }
  ospf {
    area <N> { roles: [<role>, ...] }
  }
}
```

#### bgp

**asn** — Assigns a BGP autonomous system number to each role. Every role that participates in BGP must have an entry. Use private ASN ranges for internal networks (64512–65534 for 16-bit; 4200000000–4294967294 for 32-bit).

```
asn {
  spine: 65001
  leaf:  65002
  edge:  65003
}
```

**neighbors: auto** — Automatically establishes BGP sessions between all devices whose roles are connected by a link in the topology block. This is the recommended mode. The generated configuration will contain one `neighbor` statement per peer in every adjacent role.

For fine-grained control, replace `neighbors: auto` with an explicit peer list:

```
neighbors {
  R-SPINE-1: 10.0.0.1
  R-SPINE-2: 10.0.0.2
}
```

**route-reflector** — Designates a role as the BGP route reflector. All devices in this role are generated with `bgp cluster-id 1` and treat their peers as route-reflector clients. Typically the `spine` role in a Clos topology.

#### ospf

Assigns roles to OSPF areas. Each role may appear in at most one area. Area 0 is the backbone.

```
ospf {
  area 0 { roles: [spine, leaf] }
  area 1 { roles: [edge] }
}
```

**Complete example:**

```
routing {
  bgp {
    asn {
      spine: 65001
      leaf:  65002
      edge:  65003
    }
    neighbors: auto
    route-reflector: spine
  }
  ospf {
    area 0 { roles: [spine, leaf] }
    area 1 { roles: [edge] }
  }
}
```

### Policy Block

The policy block defines named routing policies. A policy is a prioritised list of stanzas evaluated top-down in rank order — the first matching stanza determines the outcome. This is called first-match semantics.

```
policy {
  define <POLICY-NAME> {
    rank <N>: permit {
      match <expression>
      set   <expression>
      if    <condition>
    }
    rank <N>: deny {
      match <expression>
    }
  }
}
```

Policy names are conventionally written in `UPPER-KEBAB-CASE`. A program may define multiple policies; each must have a unique name.

#### rank

Assigns a numeric priority to a stanza. Lower numbers are evaluated first. Ranks must be unique within a policy (E-7). Spacing ranks in increments of 10 leaves room for future insertions.

#### permit and deny

`permit` allows matching routes to pass and optionally modifies their attributes via `set`. `deny` silently drops matching routes. A policy with no explicit catch-all `deny` implicitly permits anything not matched earlier. It is good practice to end every policy with an explicit `rank N: deny { match any }`.

#### match

Specifies which routes a stanza applies to. Exactly one match expression is allowed per stanza.

| Expression | Description |
|---|---|
| `match prefix <CIDR> le <N>` | Matches routes within the prefix up to the given length. E.g. `match prefix 10.0.0.0/8 le 24` matches any subnet of `10.0.0.0/8` with a prefix length ≤ 24. |
| `match aspath "<regex>"` | Matches routes whose BGP AS-path matches the regular expression. |
| `match community "<value>"` | Matches routes carrying the specified BGP community. |
| `match any` | Matches all routes unconditionally. Use as a catch-all. |

#### set

Modifies attributes of routes that match a `permit` stanza. Has no effect inside `deny`.

| Expression | Description |
|---|---|
| `set local-pref <N>` | Sets BGP local preference. Higher values are preferred. |
| `set metric <N>` | Sets the route metric (MED in BGP). Lower values are preferred by the receiving AS. |

#### if

An optional guard condition evaluated after the `match` expression. If false, the stanza is skipped and evaluation continues to the next rank.

| Condition | Meaning |
|---|---|
| `if neighbor.state == LIVE` | Peer is active and in full BGP session |
| `if neighbor.state == DRAINED` | Peer is administratively drained |
| `if neighbor.state == WARM` | Peer is in warm standby |

**Complete example:**

```
policy {
  define PREFER-PRIMARY {
    rank 10: permit {
      match prefix 192.168.0.0/16 le 24
      set local-pref 300
      if neighbor.state == LIVE
    }
    rank 20: permit {
      match any
      set local-pref 100
    }
    rank 30: deny {
      match prefix 0.0.0.0/0 le 32
    }
  }
}
```

Reading this policy: if the neighbor is live and the route falls within `192.168.0.0/16`, prefer it strongly (local-pref 300). If the neighbor is not live, or the route does not match that prefix, fall through to rank 20 and permit it with weaker preference (local-pref 100). Finally, drop the default route entirely.

### Intent Block

The intent block declares high-level traffic path requirements. Rather than configuring individual interfaces, you declare where traffic should flow and under what constraints — RouterLang derives the per-device implications during config generation.

```
intent {
  <INTENT-NAME>: route <label> {
    primary: <role> >> <role> >> <role>
    backup:  <role> >> <role>
    apply-policy: <POLICY-NAME>
    fault-tolerance: <N>
    scope: all
  }
}
```

Intent names are conventionally uppercase. The `route` keyword introduces a routing intent; the `<label>` after it names the traffic class (e.g. `backbone`, `datacenter`, `management`).

#### primary and backup

Path expressions describe the sequence of roles traffic must traverse, from ingress to egress. The `>>` operator separates consecutive hops.

```
primary: edge >> spine >> leaf
backup:  edge >> leaf
```

The compiler validates every path against the declared topology:

- Each role in the path must be declared (E-10).
- Every consecutive pair of roles must be connected by a link (E-11).
- The backup path must differ from the primary path (E-12).

A path must contain at least two roles. The `backup` line is optional — if omitted, no fallback is declared for this intent.

#### apply-policy

References a policy by name. The named policy must exist (E-8). During config generation, the referenced policy is emitted as a route-map on every device involved in this intent.

#### fault-tolerance

Declares the minimum number of edge-disjoint paths required. Must be 1 or greater — 0 is a semantic error (E-13). A value of 2 requires a backup path to also be declared.

#### scope

Controls which devices the intent configuration is written to. `all` applies it to every device; specific role names restrict application.

```
scope: all
scope: edge, spine
```

**Complete example with multiple intents:**

```
intent {
  CORE-TRAFFIC: route backbone {
    primary: edge >> spine >> leaf
    backup:  edge >> leaf
    apply-policy: PREFER-PRIMARY
    fault-tolerance: 2
    scope: all
  }

  MGMT-TRAFFIC: route management {
    primary: edge >> spine
    apply-policy: BLOCK-BOGONS
    fault-tolerance: 1
    scope: edge
  }
}
```

---

### Transition Block

The optional transition block describes a planned live migration between topology states. It is placed at the end of the file, after the intent block.

```
transition {
  from: <name>
  to:   <name>
  intermediate: <name>
}
```

The three fields label the source topology, target topology, and intermediate cutover state. These are free-form identifiers for documentation purposes.

```
transition {
  from: spine-v1
  to:   spine-v2
  intermediate: spine-hybrid
}
```

> **Note:** The transition block is fully parsed and included in the symbol table in the current release, but does not yet influence config generation. Support for incremental rollout planning is planned for a future version.

```
// Single-line comment

```

### Identifiers and Naming Rules

Identifiers must start with a letter (`a–z`, `A–Z`) followed by any combination of letters, digits, underscores (`_`), and hyphens (`-`).

Valid: `spine`, `R-LEAF-1`, `PREFER-PRIMARY`, `area_core`, `bgp-peer-01`

Invalid: `1spine` (starts with digit), `spine leaf` (contains space), `spine.leaf` (dot is only valid inside guard expressions)

Conventions: policy names in `UPPER-KEBAB-CASE`, role names in `lower-kebab-case`, device names following your organisation's existing hostname convention.

---

### Common Mistakes

**Referencing a role before declaring it** — all roles must be declared in the `roles` section before being used in links, ASN assignments, OSPF areas, or path expressions.

**Declaring a link without a corresponding path** — a link in the topology block does not automatically create a path in an intent. Paths must be declared explicitly, and every hop must have a corresponding link.

**Using a reserved keyword as a role name** — names like `all`, `border`, `route`, `area`, and `scope` are reserved. Using them as role names produces a parse error.

**Declaring `fault-tolerance: 2` without a backup path** — if fault tolerance is greater than 1 but no `backup` path is provided, the intent cannot be satisfied and the compiler will flag it.

**Referencing an undefined policy** — `apply-policy: MY-POLICY` in an intent requires `define MY-POLICY { ... }` to exist in the policy block (E-8).

## A Complete Example

Save the following as `my_network.rl` and run `python main.py my_network.rl --generate`.

```
topology {
  roles {
    spine  { count: 2 }
    leaf   { count: 4 }
    edge   { count: 2 }
  }
  links {
    spine -- leaf  { weight: 1 }
    spine -- edge  { weight: 2 }
    edge  -- leaf  { weight: 3 }
  }
  devices {
    spine: [R-SPINE-1, R-SPINE-2]
    leaf:  [R-LEAF-1, R-LEAF-2, R-LEAF-3, R-LEAF-4]
    edge:  [R-EDGE-1, R-EDGE-2]
  }
}

routing {
  bgp {
    asn {
      spine: 65001
      leaf:  65002
      edge:  65003
    }
    neighbors: auto
    route-reflector: spine
  }
  ospf {
    area 0 { roles: [spine, leaf] }
    area 1 { roles: [edge] }
  }
}

policy {
  define PREFER-PRIMARY {
    rank 10: permit {
      match prefix 192.168.0.0/16 le 24
      set local-pref 300
      if neighbor.state == LIVE
    }
    rank 20: permit {
      match any
      set local-pref 100
    }
    rank 30: deny {
      match prefix 0.0.0.0/0 le 32
    }
  }
}

intent {
  CORE-TRAFFIC: route backbone {
    primary: edge >> spine >> leaf
    backup:  edge >> leaf
    apply-policy: PREFER-PRIMARY
    fault-tolerance: 2
    scope: all
  }
}
```

This produces eight files under `output/my_network/` (Cisco IOS by default):

```
R-SPINE-1.cfg    R-SPINE-2.cfg
R-LEAF-1.cfg     R-LEAF-2.cfg     R-LEAF-3.cfg     R-LEAF-4.cfg
R-EDGE-1.cfg     R-EDGE-2.cfg
```

To generate the same topology for JunOS or OpenConfig, add `--vendor` — no changes to the `.rl` file are needed:

```bash
python main.py my_network.rl --generate --vendor junos       # produces .conf files
python main.py my_network.rl --generate --vendor openconfig  # produces .json files
```

## Semantic Rules

RouterLang enforces 14 rules at compile time. All errors must be resolved before configuration can be generated. Warnings are reported but do not block generation.

| Code | Severity | Rule |
|---|---|---|
| E-1 | Error | Duplicate role name |
| E-2 | Error | Role count equals zero |
| E-3 | Error | Count range has lower bound greater than upper bound |
| E-4 | Error | Link references an undeclared role |
| W-5 | Warning | Duplicate link between the same two roles (merged, not fatal) |
| E-6 | Error | Reflexive link — a role linked to itself |
| E-7 | Error | Duplicate rank within the same policy |
| E-8 | Error | Intent references an undefined policy |
| E-9 | Error | Duplicate intent name |
| E-10 | Error | Path expression references an undeclared role |
| E-11 | Error | Consecutive roles in a path have no connecting link |
| E-12 | Error | Backup path is identical to the primary path |
| E-13 | Error | `fault-tolerance` set to zero on a route intent |
| E-14 | Error | BGP ASN assigned to an undeclared role |

Diagnostics include the line number and a descriptive message. Errors are prefixed `[E-N] ERROR (line N):`; warnings are prefixed `[W-N] WARNING (line N):`.

## Config Generation

When all four validation stages pass, `--generate` produces per-device configuration files. The output format is controlled by `--vendor`.

### Cisco IOS (default — `.cfg`)

Each file contains:

- **hostname** — the device name from the `devices` section
- **OSPF** — `router ospf 1` with a deterministic router-id and the correct area assignment for this device's role
- **BGP** — `router bgp <ASN>` with one `neighbor` entry per peer device in all adjacent roles; route-reflector devices include `bgp cluster-id 1` and `route-reflector-client` per peer
- **Route-maps** — one `route-map` block per policy, with numbered sequence entries per rank; `match ip address prefix-list` and `set local-preference` are emitted from the policy block
- **Intent comments** — human-readable annotations at the end of each file describing the primary path, backup path, applied policy, and fault-tolerance requirement

Example output for `R-SPINE-1` (truncated):

```
! Auto-generated by RouterLang  [vendor: cisco]
! Device    : R-SPINE-1  |  Role: spine  |  Loopback: 10.0.1.1 (IPAM)

hostname R-SPINE-1

router ospf 1
  router-id 10.0.1.1
  network 0.0.0.0 255.255.255.255 area 0

router bgp 65001
  bgp router-id 10.0.1.1
  bgp cluster-id 1
  neighbor 10.0.3.1 remote-as 65003
  neighbor 10.0.3.1 description R-EDGE-1
  neighbor 10.0.3.1 route-reflector-client
  ...

route-map PREFER-PRIMARY permit 10
  match ip address prefix-list PL-PREFER-PRIMARY-10
  set local-preference 300
...
end
```

### JunOS (`--vendor junos` — `.conf`)

Each file uses JunOS hierarchical set-style CLI and contains:

- **system** — `host-name` and role annotation
- **protocols ospf** — router-id and area assignments
- **protocols bgp** — one `group` per peer device, typed `internal` or `external` based on ASN comparison; `cluster` configuration for route-reflector roles
- **policy-options** — named `prefix-list` blocks from `match prefix` clauses, followed by `policy-statement` entries with `term` items per rank using JunOS `accept` / `reject` syntax

### OpenConfig (`--vendor openconfig` — `.json`)

Each file is structured according to OpenConfig YANG models (RFC 7951 encoding) and contains:

- **openconfig-system:system** — hostname
- **openconfig-bgp:bgp** — global ASN, router-id, per-peer `neighbors` list with transport and route-reflector config
- **openconfig-ospfv2:ospfv2** — router-id and area assignments
- **openconfig-routing-policy:routing-policy** — `policy-definition` entries with `statement` items mapping ranks to match conditions and actions
- **_routerlang_intents** — metadata array documenting declared intents, paths, policy references, and fault-tolerance values

OpenConfig output can be pushed to devices that support NETCONF or RESTCONF with OpenConfig models loaded. It cannot be deployed via the `--deploy` flag (which uses NAPALM SSH); use your NETCONF toolchain for that format.

### Output file extensions

| Vendor | Flag | Extension |
|---|---|---|
| Cisco IOS | `--vendor cisco` | `.cfg` |
| JunOS | `--vendor junos` | `.conf` |
| OpenConfig | `--vendor openconfig` | `.json` |

When no `--ipam` file is provided, router-ids and peer IPs are derived deterministically from device names so output is always reproducible. Generated files should be reviewed by a network engineer before being applied to any live device.

## Deployment

The `--deploy` flag pushes generated configs to network devices via NAPALM. Deployment requires an `--ipam` CSV file with `mgmt_ip` addresses for every device, and a `credentials.ini` file for SSH access.

### credentials.ini format

```ini
[defaults]
username = admin
password = secret
ssh_port = 22
```

### Deployment modes

| Mode | Command | Effect |
|---|---|---|
| Simulate | *(default when `SIMULATE_DEPLOY = True`)* | Reads config files and reports a realistic diff — no SSH connection made |
| Dry run | `--deploy --dry-run` | Connects via SSH, fetches the running config, shows the diff — applies nothing |
| Live | `--deploy` | Connects, diffs, and commits the change to the device |

A timestamped deploy log is written to `./logs/` after every run, recording the diff, status, and timing for each device.

### Testing deployment locally

`fake_router.py` runs a minimal SSH stub server on local ports that responds to NAPALM connections. Use it to validate the deploy pipeline without real hardware:

```bash
python fake_router.py          # starts stub routers on ports 6001–6008
python main.py my_network.rl --generate --deploy --ipam ipam.csv
```
